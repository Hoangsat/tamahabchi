## 1. КРАТКОЕ ОПИСАНИЕ (ВИДЕНИЕ)

**Tamahabchi** — это мобильная игра, которая помогает игроку развивать реальные навыки и формировать полезные привычки через игровую механику.

**Главный цикл:**
Игрок выбирает навык → запускает режим фокуса (таймер) → получает награду (монеты + `SP` к навыку + отклик питомца) → навык растёт по уровням → паутинная диаграмма меняется → питомец сыт и счастлив → открывается новый контент

**Ключевая фишка:**  
Бесконечная паутинная диаграмма (radar chart), куда игрок может добавлять ЛЮБОЕ КОЛИЧЕСТВО НАВЫКОВ (3, 10, 20, 100 — без ограничений). Каждый навык — отдельная ось. Длина оси = `axisPercent (0–100%)`, который считается через level bands поверх `totalSP`. Диаграмма показывает ТОП-12 самых развитых навыков.

**Вторая ключевая фишка — умные боссы:**  
Боссы имеют свою паутину навыков. Если твой навык ниже, чем у босса — ты получаешь урон. Если выше — наносишь больше урона. Боссы показывают твои слабые места и мотивируют развивать паутину равномерно.

**Третья ключевая фишка — два типа ежедневных заданий:**
- Навыковые задания — привязаны к паутине («30 мин математика»)
- Простые рутины — забота о себе («выпить воду», «8 часов сна», «принять душ»)

**Эмоциональное обещание:**  
«Ты развиваешь себя в реальной жизни — твой питомец и паутина растут вместе с тобой. А боссы подсказывают, куда двигаться дальше».

### 1.1.1 Product Lock: pet XP / pet level removed from current scope

На `2026-04-11` фиксируем продуктово:

- `pet XP` и `pet level` больше не участвуют в активном gameplay loop;
- `Focus`, `Missions`, `Routines`, `Battle`, `Shop` и `Feed` не выдают опыт питомцу;
- `Room` upgrade и другие продуктовые unlock-гейты не завязаны на уровне питомца;
- прогресс игрока выражается через `coins`, `pet state` и развитие навыков через `SP`;
- поля `progression.level` / `progression.xp` могут оставаться в save schema только как legacy-совместимость;
- legacy `rewardXp` в mission/battle runtime и старые XP-gain knobs больше не являются активным runtime-контрактом.

Если ниже встречаются старые примеры с `pet XP` или `pet level`, считать их историческими заметками, а не актуальной спецификацией.

- не открываем новый большой gameplay-system поверх текущего vertical slice;

### 1.2 Override: актуальная skill progression model

Ниже в документе могут встречаться старые блоки, где skill progression описана через `percent 0..100` как основной источник истины. Для текущего кода это уже legacy-описание. Актуальная реализация на `2026-04-11` такая:

- source of truth у навыка: `totalSP`
- вычисляемые значения:
  - `level`
  - `progressInLevel`
  - `requiredSPForNextLevel`
  - `progressInLevel01`
  - `axisPercent`
- progression table:
  - `100, 160, 256, 410, 656, 1050, 1680, 2688, 4300, 6880`
- уровни: `Lv.0..Lv.10`
- `axisPercent` считается по level bands:
  - каждый завершённый уровень = `10%`
  - текущий прогресс уровня заполняет следующий `10%` сегмент
  - навык на max level даёт `100%` ось
- новый навык стартует как:
  - `Level 0`
  - `totalSP = 0`
  - `axisPercent = 0`
- `Golden / Maxed` привязаны к `Level 10`, а не к старому `percent >= 100`
- `Focus` выдаёт skill reward через `SP`
- `Mission` skill rewards тоже идут через `SP`
- старое поле `percent` оставлено только как legacy migration field для старых сейвов
- поверх свободного имени навыка теперь живёт semantic-layer `archetypeId`
- поле `icon` сохранено как compatibility/fallback visual token (`MTH`, `ART`, `MSC` и т.д.), а не как player-facing смысловой тип
- `SaveNormalizer` поднимает старые навыки по схеме `icon -> archetypeId` и нормализует некорректные/пустые значения в `general`

Правило чтения этого ТЗ:

- старые процентные формулы ниже считать legacy-спеком;
- для текущей реализации `Skills`, `Focus`, `Radar`, `Missions` и `Save/Load` приоритет имеет именно модель `SP -> Level -> Axis Percent`.

## 2. ГЛОБАЛЬНАЯ АРХИТЕКТУРА

### 2.1 Слои (строгие правила — не нарушать)

| Слой | Компоненты | Разрешено | ЗАПРЕЩЕНО |
|------|------------|-----------|------------|
| Bootstrap | GameBootstrap, SceneLoader | Загружать сцены, инициализировать менеджеры | Любую UI-логику, доступ к трансформам |
| Core | GameManager (только связывает) | Давать доступ к сервисам и runtime state | Хранить данные игрока, считать баланс |
| Systems | PetSystem, SkillSystem, MissionSystem, BattleSystem, ShopSystem | Содержать игровую логику, менять PlayerState | Обращаться к UI (кнопкам, текстам) |
| Data | PlayerData, SaveManager, Definitions | Загружать/сохранять JSON, хранить конфиги | Содержать игровую логику |
| UI | Все Panel, Button, Text, Slider | Показывать данные, отправлять команды в системы, слушать события | Считать награды, менять save напрямую |
| Events | EventBus (C# events / UnityEvents) | Передавать события между системами | Прямые ссылки между системами |

**Текущая runtime-реальность:**
- `Systems` уже являются source of truth для навыков, питомца, миссий и магазина.
- `GameManager` остаётся composition root и facade для UI, но уже не является главным единственным blocker-ом; основной остаточный риск сместился в `MissionSystem.Generation.cs`, mixed scene/runtime UI assembly и visual/mobile polish.
- `Idle v1` уже встроен в runtime через `IdleBehaviorSystem + IdleCoordinator + HUDUI`, а не как отдельный временный мод.
- `Bootstrap`, `SceneLoader` и полноценный `EventBus` остаются целевым архитектурным направлением.
- Ниже фиксируется target structure, а не точная текущая карта файлов один-в-один.

### 5.2 Правила (железобетонные)

| № | Правило |
|---|---------|
| 1 | НЕТ ОГРАНИЧЕНИЯ на количество навыков. Можно добавить 100+ |
| 2 | Удалить навык можно ТОЛЬКО если `totalSP == 0` |
| 3 | Если `totalSP > 0` → кнопка 🗑️ НЕАКТИВНА (disabled) |
| 4 | Навык растёт ТОЛЬКО от фокуса на этот конкретный навык; рост считается в `SP` |
| 5 | При достижении 100%: питомец получает перманентный бонус +5% к опыту за фокус на этот навык |
| 6 | При 100% вершина на паутине становится золотой |
| 7 | Первые 3 навыка — бесплатно. 4-й и далее — 50 монет за новый слот |

### 5.3 Формула роста навыка

```csharp
int CalculateSkillPointsFromFocusDuration(float durationSeconds) {
    if (durationSeconds <= 0f) {
        return 0;
    }

    return Mathf.Max(0, Mathf.FloorToInt(durationSeconds / 60f));
}
```

**Примеры:**
- 15 минут → `15 SP`
- 25 минут → `25 SP`
- 60 минут → `60 SP`
- дальнейший UI-прогресс считается через `SP -> Level -> axisPercent`

## 6. ПАУТИННАЯ ДИАГРАММА (RADAR CHART) — ТОП-12

### 6.1 Логика отображения

| Количество навыков | Что показывает диаграмма |
|-------------------|--------------------------|
| 0–2 | Сообщение: «Добавьте хотя бы 3 навыка» |
| 3–12 | Все навыки (3–12 осей) |
| 13+ | ТОП-12 по проценту (при равных — по дате последнего фокуса) |

### 6.2 Математика отрисовки

```
Угол между осями = 360° / количество_отображаемых_навыков
Радиус диаграммы = 200 пикселей (в UI)
Центр = (x, y) = (300, 300)

Для i-го навыка:
    angle = i * angleStep * Mathf.Deg2Rad
    pointRadius = (skill.percent / 100f) * radius
    x = centerX + pointRadius * Mathf.Cos(angle)
    y = centerY + pointRadius * Mathf.Sin(angle)
```

### 6.3 Визуальные параметры

| Элемент | Цвет | Толщина | Прозрачность |
|---------|------|---------|--------------|
| Оси (линии от центра) | #888888 | 2px | 1.0 |
| Кольца (33%, 66%, 100%) | #444444 | 1px | 0.5 |
| Полигон (паутина) | #00AAFF | заполнение | 0.4 |
| Контур полигона | #00AAFF | 3px | 1.0 |
| Точки на вершинах | #00AAFF | 8px диаметр | 1.0 |
| Золотая вершина (100%) | #FFD700 | 12px диаметр | 1.0 + пульсация |
| Текст названия навыка | #FFFFFF | шрифт 14 | 1.0 |

### 6.4 Реализация в Unity (текущая)

**Текущая реализация:** собственный UGUI-компонент `RadarChartGraphic`, без внешней chart-зависимости.

**Почему выбран свой компонент:**
- нужен только radar, а не целая chart-библиотека;
- полный контроль над кольцами, осями, полигоном, outline и точками;
- проще держать `ТОП-12`, анимацию и золотые вершины при `100%`;
- в проекте нет внешней зависимости, которую нужно поддерживать.

**Текущее устройство:**
- `SkillsPanelUI` выбирает `TOP-12` навыков;
- `RadarChartGraphic` рисует сетку, оси, паутину и точки;
- обновление идёт через передачу нормализованных значений и цветов.

### 6.5 Обновление диаграммы

```csharp
public void UpdateRadarChart() {
    List<Skill> topSkills = skillSystem.GetTopSkills(12);
    var normalizedValues = topSkills.Select(skill => skill.percent / 100f).ToList();
    var markerColors = BuildPalette(topSkills.Count);
    radarChartGraphic.SetValuesAnimated(normalizedValues, markerColors, changedIndex);
}
```

### 8.2 Техническая реализация таймера

```csharp
public class FocusTimer : MonoBehaviour {
    private float remainingSeconds;
    private bool isRunning;
    private Coroutine timerCoroutine;
    
    public void StartTimer(int minutes) {
        remainingSeconds = minutes * 60;
        isRunning = true;
        timerCoroutine = StartCoroutine(TimerCoroutine());
    }
    
    private IEnumerator TimerCoroutine() {
        float lastTime = Time.realtimeSinceStartup;
        while (isRunning && remainingSeconds > 0) {
            float now = Time.realtimeSinceStartup;
            float delta = now - lastTime;
            lastTime = now;
            remainingSeconds -= delta;
            UpdateUI(remainingSeconds);
            yield return null;
        }
        if (remainingSeconds <= 0) {
            OnTimerComplete();
        }
    }
    
    public void Pause() { isRunning = false; }
    public void Resume() { /* перезапуск корутины */ }
    public void CompleteEarly() { OnTimerComplete(); }
}
```

## 10. БОЕВАЯ СИСТЕМА (RPG LITE) — УМНЫЕ БОССЫ

### 10.0 Battle design v1 (актуальный override)

Ниже в разделе ещё могут встречаться старые блоки про прямое сравнение паутин игрока и босса по каждой оси. Для `Battle v1` это считается legacy-идеей и **не** является основной балансной моделью.

Текущая целевая модель для первой версии боя:

- `SP = реальная сила навыка`
- `Level = UX / readability`
- `Boss Power` считается по **cumulative totalSP** для target level
- `Player Battle Power` считается по `top 3 skills by axisPercent`
- паутина босса в `v1` — это визуальный слой, а не основной расчёт боевой силы

### 10.0.1 Сила игрока

Используется формула:

```text
Player Battle Power = average(totalSP of top 3 skills by axisPercent)
```

Почему именно так:

- игрок не штрафуется за добавление новых навыков с нулевым прогрессом
- формула не ломается при 3 навыках и при 20 навыках
- `axisPercent` остаётся важным как способ выбрать strongest combat skills
- реальная сила берётся из `totalSP`, а не из декоративной оси

### 10.0.2 Сила босса

Каждый босс имеет `Boss Power`, равный суммарному `totalSP`, нужному для достижения целевого уровня.

| Босс | Целевой уровень | Boss Power |
|------|----------------:|-----------:|
| Boss 1 | Lv1 | 100 |
| Boss 2 | Lv2 | 260 |
| Boss 3 | Lv3 | 516 |
| Boss 4 | Lv4 | 926 |
| Boss 5 | Lv5 | 1582 |
| Boss 6 | Lv6 | 2632 |
| Boss 7 | Lv7 | 4312 |
| Boss 8 | Lv8 | 7000 |
| Boss 9 | Lv9 | 11300 |
| Boss 10 | Lv10 | 18180 |

### 10.0.3 Правило победы

```text
if Player Battle Power >= Boss Power
    player wins
else
    player loses
```

### 10.0.4 Практические правила v1

- battle unlock рекомендуется при `минимум 3 tracked skills`
- на валидный бой тратится энергия питомца, но победа не даёт `pet XP`
- при равенстве `axisPercent` для top-3 ranking использовать:
  - `totalSP`
  - затем `lastFocusDate` как tie-break
- `Boss Power` хранить в data table / definitions, а не разбрасывать по коду

### 10.0.5 Паутина босса в v1

Паутина босса используется как визуальный элемент, mood/fantasy и подсказка по теме босса, но не влияет напрямую на основной расчёт победы в `Battle v1`.

### 10.1 Новая логика — сравнение паутин

Главная идея: У каждого босса есть своя «паутина навыков» — набор навыков, в которых он силён или слаб.

- Если твой навык выше, чем у босса → ты наносишь бонусный урон
- Если твой навык ниже, чем у босса → босс наносит дополнительный урон тебе
- Если навыков нет в паутине босса — нейтрально

## 20. ОТВЕТЫ НА ЧАСТЫЕ ВОПРОСЫ РАЗРАБОТЧИКА

**Q: Как часто сохранять?**
A: После каждого действия: фокус, покупка, кормление, задание, битва, рутина. А также при паузе и каждые 60 секунд.

**Q: Как считать «пропущенный день»?**
A: Сравниваем lastActiveTime с текущим временем. Если разница > 24 часов — применяем штраф к голоду.

**Q: Можно ли добавить БОЛЬШЕ 12 навыков?**
A: ДА. НЕТ ОГРАНИЧЕНИЙ. Диаграмма показывает ТОП-12, список показывает все.

**Q: Что, если игрок хочет удалить навык с 80%?**
A: НЕЛЬЗЯ. Кнопка удаления неактивна. В будущем — «перековка» за монеты (v1.5).

**Q: Как работают умные боссы?**
A: У каждого босса своя паутина навыков. Сравнивается с паутиной игрока. Если твой навык ниже — получаешь больше урона. Если выше — наносишь больше.

**Q: Чем рутина отличается от навыкового задания?**
A: Рутина не требует фокуса и таймера. Игрок просто отмечает выполнение и получает награду. Подходит для простых привычек (вода, сон, душ).

**Q: Можно ли создать свою рутину?**
A: ДА. Игрок может создать любую рутину с названием и выбрать награду (монеты, уход/энергия, настроение, прогресс навыка).

**Q: Как босс подсказывает, что развивать?**
A: В интерфейсе битвы отображается сравнение паутин. Если у босса навык выше, показывается совет: «Босс сильнее в Математике! Развивай её!».

**Q: Как реализована паутина сейчас?**
A: Своим UGUI-компонентом `RadarChartGraphic`, который рисует оси, кольца, полигон, outline и точки без внешнего chart-ассета.
